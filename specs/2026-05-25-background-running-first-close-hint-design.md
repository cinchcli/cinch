# Background-running first-close hint (macOS)

**Date:** 2026-05-25
**Component:** `apps/desktop` (Tauri v2 backend + React frontend)
**Status:** Designed (not yet implemented)
**Baseline:** branch `main` @ `f6f732d`

## Goal

The Cinch desktop app is a background menu-bar agent: no Dock icon, hidden
from Cmd+Tab, and closing the window (close box / Cmd+W / Cmd+Q) hides it
while the app keeps running in the menu bar. This is correct behavior, but
**silent** — a new user who closes the window sees it vanish with no Dock
icon to reassure them, and assumes they quit the app. They didn't.

Tell the user, **once**, the first time the window is dismissed: the app is
still running in the menu bar, here's how to find it, and here's how to
actually quit if that's what they wanted.

## Background: what already exists

- Window starts hidden — `tauri.conf.json` window has `"visible": false`.
- App is a macOS Accessory agent — no Dock icon, hidden from Cmd+Tab
  (`window_manage.rs:171` `configure_activation_policy`).
- Close box / Cmd+W → `lib.rs:301` `CloseRequested` → `prevent_close()` +
  `window.hide()`.
- Cmd+Q → custom menu item `HIDE_WINDOW_ID` (`app_menu.rs:24`) →
  `handle_menu_event` (`app_menu.rs:88`) → hides every webview window.
- Real quit is the tray's "Quit Cinch" → `app.exit(0)`, which passes through
  the `ExitRequested { code: Some(0) }` guard (`lib.rs:533`).
- A settings key/value store exists: `Database::get_setting` /
  `set_setting` (`store/db/settings.rs:6,20`).
- A polished custom dialog component exists: `ConfirmDialog.tsx` (native /
  legacy dialogs are forbidden by the design system).
- Typed Tauri command + event bridge via tauri-specta (`bindings.ts`,
  regenerated by `cargo test export_bindings -- --ignored`).

There is **no** signal at the moment of first dismissal. That is the gap.

## Decisions (from brainstorming)

1. **Trigger:** First time the window is dismissed (close box / Cmd+W / Cmd+Q).
   That is the moment of surprise ("wait, didn't I quit?").
2. **Delivery:** A one-time, in-window dialog shown *before* the window
   hides — not an OS notification (less likely to be missed, lets us offer
   a real Quit choice, matches the app's custom design).
3. **Quit option:** The dialog offers "완전 종료 (Quit)" in addition to
   "메뉴 막대로 유지 (Keep in menu bar)", because some users press close
   intending to quit.
4. **Custom dialog, not `ConfirmDialog` as-is:** `ConfirmDialog` focuses its
   secondary button and maps Esc → cancel, designed for *destructive*
   confirmations. Here the safe default is "keep running" — Esc and Enter
   must resolve to keep, and "keep" must be the visually prominent action.
   A small purpose-built component gets the emphasis and key semantics right.
5. **One-time:** Gated by a persisted flag; no "don't show again" checkbox —
   either button dismisses it permanently.

## Architecture & data flow

```
User dismisses window (close box / Cmd+W / Cmd+Q)
        │ (Rust)
        ▼
request_dismiss(app):
   seen = should_prompt( db.get_setting("background_hint_seen") )
   ├─ already seen → main_window.hide()                  # quiet, as today
   └─ not seen     → prevent_close + emit BackgroundHint  # keep window up
        │ (Frontend: App.tsx listens events.backgroundHint)
        ▼
   BackgroundHintDialog shown over the still-visible window
   ├─ "메뉴 막대로 유지" / Enter / Esc → resolveBackgroundHint(false)
   └─ "완전 종료"                       → resolveBackgroundHint(true)
        │ (Rust command)
        ▼
resolve_background_hint(quit):
   db.set_setting("background_hint_seen", "1")   # both paths mark it seen
   quit ? app.exit(0) : main_window.hide()
```

Key design point: the flag is read **in Rust**, synchronously inside the
window-event handler, before deciding whether to hide. So it must live in
the settings DB (`background_hint_seen`), not frontend `localStorage`.

## Components

### Rust

- **`should_prompt(flag: Option<&str>) -> bool`** (pure, in `window_manage.rs`)
  — returns `true` only when the flag is absent/not `"1"`. The only unit-
  testable logic; isolated for that reason.
- **`request_dismiss(app: &AppHandle)`** (`window_manage.rs`) — the single
  source of truth for a user-initiated dismissal. Reads the flag via the
  managed `Arc<Database>` state; hides the main window if seen, otherwise
  emits the `BackgroundHint` event (and does **not** hide, so the dialog is
  visible). Used by both dismissal paths.
- **`CloseRequested` handler** (`lib.rs:301`) — change `window.hide()` to
  `request_dismiss(window.app_handle())` (still after `api.prevent_close()`).
- **Cmd+Q handler** (`app_menu.rs:88` `handle_menu_event`) — change the
  hide-all loop to `request_dismiss(app)` so Cmd+Q funnels through the same
  gate. (First dismissal via Cmd+Q also shows the dialog.)
- **`BackgroundHint`** — new `tauri_specta::Event` (unit payload).
- **`resolve_background_hint(quit: bool)`** — new command: sets the flag,
  then `app.exit(0)` (quit) or hides the main window (keep). Registered in
  the specta builder; bindings regenerated.

### Frontend

- **`BackgroundHintDialog.tsx`** — new component modeled on
  `ConfirmDialog.tsx`'s styling, with:
  - Title: "Cinch는 메뉴 막대에서 계속 실행됩니다" (Cinch keeps running in the menu bar)
  - Body: where to find the menu-bar icon / how to re-open (Cmd+Shift+W).
  - **Primary / initial focus / Enter / Esc → "메뉴 막대로 유지"** (safe default).
  - **Quiet secondary → "완전 종료"**.
- **`App.tsx`** — subscribe to `events.backgroundHint.listen` (cleanup on
  unmount, matching existing listener patterns); on event, open the dialog.
  Buttons call `commands.resolveBackgroundHint(quit)` then close the dialog.

## Edge cases

- **Programmatic hides are excluded.** Paste-and-hide (`commands/clips/misc.rs:99`)
  and overlay hides (`commands/window.rs:292`) call `window.hide()` directly
  and must continue to — they are deliberate flows, not "user quit" gestures.
  Only `CloseRequested` and Cmd+Q route through `request_dismiss`.
- **First launch:** window starts hidden; flag absent. The dialog only
  appears once the user has opened the window and then dismisses it — never
  before they've seen the app.
- **Quit before acknowledging:** clicking "완전 종료" sets the flag too, so a
  user who quits via the dialog won't be re-prompted on the next run's first
  close (they've been informed).
- **Multiple/secondary webview windows:** Cmd+Q today hides *every* webview
  window; the only persistent one is `main` (the snap overlay is transient and
  self-dismisses). `request_dismiss` therefore targets `main` — both for the
  hint (the dialog lives in `main`) and for the seen-path hide. Confirm during
  implementation that no other long-lived window relies on the old hide-all.

## Out of scope (YAGNI)

- No OS notification path (the dialog is the chosen delivery).
- No "don't show again" checkbox (the dialog is inherently one-time).
- No change to the tray menu, global shortcut, or the existing quit guard.
- No first-launch / Dock-flash changes — unchanged from the menu-bar-agent spec.
- Non-macOS: the dismissal paths and the flag are platform-neutral; the hint
  text assumes a menu-bar agent (current shipping target is macOS).

## Testing

- **Rust unit:** `should_prompt` — `None`/`Some("")`/`Some("0")` → prompt;
  `Some("1")` → don't.
- **Frontend (vitest):** mounting with a fired `backgroundHint` event renders
  `BackgroundHintDialog`; "메뉴 막대로 유지" calls `resolveBackgroundHint(false)`;
  "완전 종료" calls `resolveBackgroundHint(true)`; Esc resolves to keep.
- **Manual (macOS):** fresh state (clear `background_hint_seen`) → first close
  shows the dialog; "유지" hides + app survives; second close hides silently;
  reset flag, first close, "완전 종료" quits the process. Repeat with Cmd+Q.

## Risks / notes

- `app.exit(0)` from the command must pass through the `ExitRequested`
  guard (`lib.rs:533`) like the tray quit does — verify during implementation.
- The custom dialog must not reuse `App.tsx`'s `S.dialog` styles (forbidden
  by PATTERNS.md, per the `ConfirmDialog.tsx` header); model it on
  `ConfirmDialog`'s own styling instead.
- `lefthook` pre-commit runs `rust-fmt`/clippy and the desktop test suite;
  keep `cargo fmt` clean and regenerate `bindings.ts` before committing.
